# M2 — UX Parity: Implementation Specification

Status: **ready for implementation** (approved plan: [PLAN.md](./PLAN.md),
issue [#7](https://github.com/freelensapp/freelens-for-claude-extension/issues/7);
M0 and M1 delivered and confirmed working by the maintainer).

Success criterion: right-click a failing Pod and choose "Ask Claude" — the
chat page opens with the analysis request already sent; while Claude works,
tool calls appear as collapsible cards; the status strip shows cumulative
token usage and lets the user switch the model to `haiku` mid-conversation;
if the turn fails, the error message offers a Retry button that re-runs it;
and with the new "Require approval before reading pod logs" preference
enabled, a `kube_pod_logs` call pauses on the same approval card mutations
use.

## Scope

In scope (the M2 milestone from PLAN.md, plus the two items M1 explicitly
deferred here):

1. Token usage display: cumulative per-session counter in the status strip,
   plus a "conversation compacted" notice driven by Claude Code's native
   compaction.
2. Retry on failure: failed turns render an error item in the transcript
   with a Retry button that re-runs the turn without duplicating the user
   bubble.
3. Model picker: per-cluster model selection over the Claude Code aliases,
   switchable mid-conversation without restarting the session.
4. Tool-call cards: the M1 one-line tool notices become collapsible cards
   pairing each call with its result and showing a running state.
5. "Ask Claude" on resources: `kubeObjectMenuItems` entries that open the
   chat pre-filled with an analysis prompt for the selected object.
6. Preferences page (`appPreferences`): pod-logs approval and tail-lines
   defaults (deferred from M1), custom agent rules, a Claude Code executable
   path override, and a default model.

Out of scope (deferred per PLAN.md): user MCP-server passthrough, Claude
Code subagents, prompt shortcuts/slash commands, and the
Kubectl-via-Freelens-terminal tool (all M3). Also consciously not in M2:

- Per-cluster "always allow" tool lists — research showed the original
  extension has none (its only bypass is the transient accept-all toggle,
  which M1 already ships as the `acceptAll` mode). Dropped from the parity
  backlog; may return as an M3 nicety.
- The original's circular token-capacity gauge, ~4-chars-per-token
  estimation, and self-managed compaction threshold — all exist only
  because the original had to compact conversations itself. Claude Code
  compacts natively; M2 surfaces that as a transcript notice instead.
- The original's "Available Tools" informational panel and its
  reasoning-details fold. Both are candidates for M3 polish.

## Parity research notes

Findings from `freelensapp/freelens-ai-extension@main` that shape this
spec (deviations are deliberate and called out):

- **Token counter** (`text-input.tsx`, `token-usage.ts`): cumulative
  per-session `in:<n> (cached:<n>) + out:<n>` next to the model select,
  reset when the chat is cleared, persisted across restarts. M2 matches
  this format and placement, dropping only the appended USD cost (PLAN
  D8).
- **Retry** (`chat-service.ts` `retry`, `message.tsx`): errors become
  transcript messages carrying a retry context; retrying re-runs the agent
  with the same text and deliberately does not re-add the user bubble;
  stale error messages are cleared on the next user action. M2 matches
  this behavior.
- **Model picker** (`text-input.tsx`): a plain select in the input bar;
  switching mid-conversation just changes the model used for the next
  send, no confirmation and no session reset. M2 matches, with the option
  list fixed to the Claude Code aliases instead of a user-managed model
  list.
- **Tool calls are invisible in the original**: its agent service never
  yields tool-call or tool-result chunks; the user sees only streamed
  prose plus approval interrupts. M2 deliberately deviates — PLAN M2
  promises tool-call cards, the M0/M1 UI already shows one-line tool
  notices, and visible tool use is the established Claude Code UX. The
  cards below are an upgrade, not parity.
- **"Explain" menu** (`index.tsx`, `menu-entry-hook.tsx`): registered only
  for `Event`/v1 objects; sends the event message wrapped in "Could you
  explain this message?" through a non-persisted store observable, then
  navigates to the chat page. M2 keeps that mechanism (pending prompt +
  navigate) and extends registration to common workload kinds, since our
  tools can fetch live state that the original had to paste in.
- **Preferences** (`preferences-page.tsx`, `preferences-store.ts`): mostly
  provider/API-key machinery (dropped per PLAN D10). The provider-neutral
  items are `podLogsRequireApproval` (default **true**), `podLogsTailLines`
  (default 1000), and `customAgentRules` (free text appended to the system
  prompt). M2 carries those three over and adds two Claude-specific ones
  (executable path override, default model). The original commits textarea
  drafts to the store only on blur to avoid caret jumps — copy that.

## Dependencies

None new. Usage numbers, model switching, and compaction boundaries all
come from the already-bundled Agent SDK; the cards and preferences are
plain React/SCSS.

Relevant SDK surface (verified against `@anthropic-ai/claude-agent-sdk`
0.3.204):

- `SDKResultMessage.usage` (`input_tokens`, `output_tokens`,
  `cache_read_input_tokens`, `cache_creation_input_tokens`) — per-turn
  totals on every `result` message.
- `Query.setModel(model?: string)` — switches the model of a live session;
  `undefined` restores the default.
- `system`/`init` messages report the resolved `model` id.
- `system`/`compact_boundary` messages report native compaction with
  `compact_metadata.trigger` and `pre_tokens`.

## Protocol additions (`src/common/protocol.ts`)

All additions are optional fields or new event types, so transcripts
persisted by M1 replay unchanged.

New SSE event types in `SessionEventMap` (both added to
`PERSISTED_EVENTS` so replay restores them):

| Event | Payload |
| --- | --- |
| `usage` | `{ inputTokens, cachedInputTokens, outputTokens }` |
| `compaction` | `{ trigger: "manual" \| "auto", preTokens: number }` |

- `usage` carries the **per-turn** numbers from the SDK `result` message:
  `inputTokens` = `usage.input_tokens` + `usage.cache_creation_input_tokens`,
  `cachedInputTokens` = `usage.cache_read_input_tokens`, `outputTokens` =
  `usage.output_tokens`. The renderer sums the events it has seen, which
  makes the counter cumulative per session and automatically reset by New
  chat (the transcript is deleted) — the same lifecycle as the original.
- `compaction` is emitted for the SDK `system`/`compact_boundary` message.

Changed events:

| Event | Change |
| --- | --- |
| `tool_call` | + `callId?: string` (the SDK `tool_use` block id) |
| `tool_result` | + `callId?: string` (the block's `tool_use_id`) |
| `session_meta` | + `model?: string` (selected alias; absent = Claude Code default), + `resolvedModel?: string` (the id reported by `init`) |
| `error` | + `canRetry?: boolean` |

`callId` is optional because replayed M1 transcripts do not have it; the
renderer renders an unpaired call or result as the M1-style one-line
notice.

New request bodies and routes:

| Route | Body |
| --- | --- |
| `POST /clusters/:id/model` | `{ model: string \| null }` (`null` = default) |
| `POST /clusters/:id/retry` | none |

`MODEL_CHOICES = ["sonnet", "opus", "haiku"]` is exported from
`protocol.ts` as the single source of truth for route validation and the
picker options; `null`/absent means "Claude Code default" and is rendered
as `Default` in the picker. `POST .../model` answers `400` for anything
else. `POST .../retry` answers `409` when there is nothing to retry or a
turn is in flight, `202` otherwise.

## Main process

### Usage and compaction (`session-manager.ts`)

- In `handleMessage`, case `result`: read `message.usage` and emit a
  `usage` event with the mapping above before `turn_complete`. Missing
  usage (defensive) emits nothing.
- New case: `system` with `subtype === "compact_boundary"` emits
  `compaction` from `compact_metadata`.

### Model selection

- `ChatSessionEntry` (`session-store.ts`) gains `model?: string`,
  persisted like the permission mode; `ChatSessionState` gains
  `writeModel(clusterId, model | undefined)`.
- `ClusterSession` keeps the selected model in memory (initialized from
  the store or, when unset, from the `defaultModel` preference at session
  construction). `startQuery` passes `model` in the options when set.
- `setModel(model | undefined)`: store in memory, persist, call
  `queryHandle?.setModel(model)` when a query is live (best-effort — an
  error is surfaced as a non-fatal `error` event), and emit
  `session_meta` reflecting the new selection. Mode parity: switching
  never restarts the query and never resets the conversation.
- The `system`/`init` handler captures `message.model` and includes it as
  `resolvedModel` in subsequent `session_meta` events, so the UI can show
  what "Default" actually resolves to.
- `SessionManager.setModel(clusterId, model)` mirrors
  `setPermissionMode`, creating the session state if needed.

### Retry

- `ClusterSession` records `lastUserText` on every `sendMessage`. Errors
  emitted while a turn is in flight (the `consume` catch path and the
  `result`-with-error path) set `canRetry: true` when `lastUserText`
  exists; errors from `start()` failures (Claude Code missing) keep
  `canRetry` unset — Retry cannot fix those.
- `retry()`: refuse (return a "nothing to retry" marker for the 409) when
  `working` or `lastUserText` is unset. Otherwise re-push `lastUserText`
  into the input queue and set working — **without** emitting a new
  `user_message` event, matching the original ("the original user prompt
  is already in the history, so retrying must not re-add it"). If the
  query died, `startQuery` is rebuilt first (the existing resume-retry
  machinery already covers the restart path).

### Pod-logs approval and tail-lines preference

- `kube_pod_logs` is removed from the `allowedTools` list passed to the
  SDK, so `canUseTool` now fires for it on every call. The callback:
  - preference off: allow immediately, echoing the input (exactly the M1
    read-only behavior, minus the SDK short-circuit);
  - preference on: route through the broker like a mutating tool, but via
    a read-tool path that `readOnly` mode does **not** deny — reading
    logs is permitted in a read-only chat, it just needs consent.
    `acceptAll` auto-approves with the request/resolved event pair, as
    for mutations.
- `permission-broker.ts` gains that read-tool path (e.g. a
  `requiresConsent` flag on the decide call); everything else — request
  ids, SSE events, resolution, deny-on-interrupt — is shared with the
  mutating path.
- `approval.ts` `describeApproval` gains a `kube_pod_logs` case: action
  title `READ POD LOGS`, `proposedValue` = the tool input, no target (no
  backup needed for a read).
- The preference is read live inside the callback (not baked into query
  options), so toggling it applies to the next tool call without a new
  session.
- `podLogsTailLines` replaces the hardcoded `DEFAULT_TAIL_LINES = 200` as
  the default when the model does not pass `tailLines`: `mcp-server.ts`
  accepts a getter and threads it into `runPodLogs`. Default value 1000
  (parity with the original), still capped by `LOG_BYTE_CAP`.

### Custom agent rules and path override

- `session-manager.ts` appends the `customAgentRules` preference (when
  non-empty) to the `systemPrompt.append` text at `startQuery` time,
  separated as a clearly-delimited "Additional user rules" paragraph.
  Like the original, rules apply from the next session start (new chat or
  restart); the preferences page says so in its hint.
- `detect.ts` accepts an optional explicit path: when the
  `claudeCodePath` preference is set, detection validates exactly that
  path (executable + `--version`) and reports its failure as the
  detection error instead of falling through to auto-detection — a
  misconfigured override must be visible, not silently ignored.
  `main/index.ts` passes the preference into both `detectClaudeCode`
  call sites.

### Bridge routes (`server.ts`)

- `POST /clusters/:id/model`: validate against `MODEL_CHOICES` (or
  `null`), then `sessionManager.setModel`; answer `{ model }`.
- `POST /clusters/:id/retry`: `sessionManager.retry(clusterId)`; `202`
  `{ accepted: true }` or `409` `{ error: "Nothing to retry" }`.

## Preferences store (`src/common/preferences-store.ts`)

`PreferencesStore` (`Common.Store.ExtensionStore`, config name
`preferences-store`), created by both processes like the `BridgeStore`:

```text
podLogsRequireApproval: boolean   # default true (parity with the original)
podLogsTailLines: number          # default 1000
customAgentRules: string          # default ""
claudeCodePath: string            # default "" = auto-detect
defaultModel: string              # default "" = Claude Code default
```

A narrow read-only interface (`PreferencesState`, mirroring
`ChatSessionState`) is what the session manager and detection depend on,
so tests substitute a plain object.

## Renderer

### Status strip (`chat-view.tsx`)

- Left side gains the token counter once any `usage` event has arrived:
  `in:<n> (cached:<n>) + out:<n>` with `toLocaleString("en-US")` grouping
  — the original's exact format, minus cost — with a tooltip "Tokens used
  this session (input, cached input, output). Resets when the chat is
  cleared." The reducer sums `usage` events into
  `state.usage: { input, cached, output }`.
- A `compaction` event appends a transcript notice item ("Conversation
  compacted to save context"), styled like the existing tool notices.
- Model select next to the mode select: options `Default` (empty value) +
  `MODEL_CHOICES`; value from `session_meta.model`; `onChange` calls
  `client.setModel(clusterId, value || null)`. When
  `session_meta.resolvedModel` is present, the `Default` option label
  becomes `Default (<resolvedModel>)`.

### Error items and Retry

- `error` events with `canRetry` become transcript items (reducer:
  append `{ kind: "session_error", message, kind, canRetry }`) instead of
  only setting the banner; the banner remains for non-retryable bridge
  failures (send failed, mode change failed).
- The Retry button renders only on the **last** transcript item, only
  while not `working` — the original's "stale errors lose their retry"
  rule, implemented append-only. Clicking calls `client.retry(clusterId)`
  and relies on the resulting `status: working` event to update the UI.

### Tool cards (`tool-card.tsx`)

- The reducer pairs events by `callId`: `tool_call` appends a card item
  `{ kind: "tool", callId, toolName, input, result? }`; a `tool_result`
  with a matching `callId` fills `result` in place. Events without
  `callId` (replayed M1 transcripts) keep the M1 one-line notices.
- Card UI: a collapsed single-line header — short tool name
  (`kube_resources`, without the `mcp__...` prefix, which the main
  process already emits unqualified via the M1 events) plus a compact
  argument summary (namespace/name/kind when present) and a state
  indicator: an animated "running" dot while `result` is unset, nothing
  once resolved. Clicking expands two blocks: the input rendered as
  YAML-ish `<pre>` (reuse the permission dialog's code-block styling) and
  the result summary text. Permission dialogs are unchanged and continue
  to render as their own items.

### "Ask Claude" menu entries

- `pending-prompt.ts`: a module-scoped holder with `set(prompt)` /
  `consume(): string | undefined` (single renderer process; no store
  round-trip needed — the original used a store observable only because
  its chat lived behind the same mechanism).
- `menu-entry.tsx`: a `MenuItem` component (props
  `Renderer.Component.KubeObjectMenuProps`) with the extension icon and
  title "Ask Claude". `onClick` builds the prompt, calls
  `pendingPrompt.set(...)`, and navigates to the chat page via the
  extension instance's `navigate(PAGE_ID)` (the registration closure in
  `index.tsx` has `this`).
- Registrations in `index.tsx` (`kubeObjectMenuItems`), one per kind:
  - `Event`/`v1` — parity with the original's "Explain": prompt "Could
    you explain this message?" plus the event message, shown only when
    the event has a message.
  - `Pod`, `Deployment`, `DaemonSet`, `StatefulSet` (`apps/v1`),
    `Service`, `Node` — prompt "Analyze `<kind> <namespace>/<name>` in
    this cluster: check its current state, recent warning events and (for
    workloads) pod logs, and summarize any problems." No YAML is
    embedded: the model fetches live state through the `kube_` tools,
    which is fresher than pasting a snapshot.
- `chat-view.tsx` consumes the pending prompt in the mount effect (after
  the SSE stream opens) and sends it through the normal `send` path, so
  it appears as a regular user bubble (parity: the original shows the
  full explain prompt as the user message).

### Preferences page (`preferences.tsx`)

Registered in `index.tsx` via `appPreferences` with title
"Freelens for Claude" (one entry; `Hint` renders the empty span, as in
the original). Controls, top to bottom, all bound to `PreferencesStore`:

1. Claude Code executable path (`Input`, monospace; hint: "Absolute path
   to the claude binary. Leave empty to auto-detect.").
2. Default model (`Select`: Default + `MODEL_CHOICES`; hint: "Used for
   clusters that have not picked a model in the chat.").
3. Custom agent rules (`textarea`; hint: "Extra rules appended to the
   system prompt at the start of every new session."). Commit the draft
   to the store on blur, not per keystroke (caret-jump avoidance, as in
   the original).
4. Require approval before reading pod logs (`Switch`; hint: "Pod logs
   can contain secrets or personal data. When enabled, the agent asks for
   confirmation before reading container logs.").
5. Default tail lines (number `Input`; hint: "Lines read from the end of
   the log when the agent does not request an amount.").

Use the host's `Renderer.Component` form controls (`Input`, `Switch`,
`Select`) so the page matches Freelens preferences styling.

## Testing and validation

Unit tests (vitest, extending the M1 files and stub pattern):

- protocol: encode/decode round-trip for `usage` and `compaction`;
  `tool_call`/`tool_result` with and without `callId`; `session_meta`
  with model fields.
- session manager (via the existing seams): `result` message emits
  `usage` with the documented field mapping; `compact_boundary` emits
  `compaction`; retry refuses while working and re-pushes without a new
  `user_message` event; errors set `canRetry` only after a user turn.
- broker: consent-required read tool prompts in `readOnly` mode instead
  of being denied; `acceptAll` auto-approves it; preference off bypasses
  the broker entirely.
- bridge: model route validation (alias, `null`, garbage), retry route
  `202`/`409`.
- stores: `ChatSessionEntry.model` round-trip; `PreferencesStore`
  defaults and the `PreferencesState` view.
- pod logs: configurable tail-lines default reaches `readNamespacedPodLog`.
- detection: explicit override path used when set; invalid override
  reported as the error, not silently skipped.

Validation gates unchanged (AGENTS.md): `pnpm lint:fix &&
pnpm type:check && pnpm test:unit` and `pnpm build`; everything must stay
green with Claude Code absent.
