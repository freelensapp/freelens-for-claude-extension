# M0 — Proof of Concept: Implementation Specification

Status: **delivered** — implemented, merged, and confirmed working by the
maintainer (approved plan: [PLAN.md](./PLAN.md), issue
[#7](https://github.com/freelensapp/freelens-for-claude-extension/issues/7)).
The next milestone is specified in [M1.md](./M1.md).

Success criterion: with a Claude Code installation authenticated by the
user (subscription login), open the "Freelens for Claude" page on a
connected cluster, ask *"what pods are failing in this cluster and why?"*,
and receive an answer grounded in read-only tool calls.

## Scope

In scope:

1. Claude Code detection and an onboarding panel when it is missing.
2. Local HTTP bridge (main process): `node:http` on `127.0.0.1`, ephemeral
   port, per-launch bearer token, SSE event stream.
3. One Claude Agent SDK session per cluster, in memory for the app run.
4. Three read-only MCP tools: resource list/get, pod logs, warning events.
5. Minimal per-cluster chat page: message list with markdown rendering,
   input box, working indicator, stop button, new-chat action.

Out of scope for M0 (deferred to M1+ per the approved plan): mutating
tools, approval dialogs, `permissionMode` preferences, session persistence
and `resume` across restarts, model picker, token usage display, retry,
tool-call cards, "Ask Claude" context menu, user MCP passthrough, Kubectl
tool via Freelens terminal. Nothing in M0 may require a permission prompt:
every tool is read-only and auto-allowed.

## Dependencies

Add to `dependencies` (bundled):

| Package | Purpose |
| --- | --- |
| `@anthropic-ai/claude-agent-sdk` | agent runtime, spawns the user's Claude Code |
| `zod` | MCP tool input schemas |
| `@kubernetes/client-node` | cluster access for tools (main process only) |
| `react-markdown` + `remark-gfm` | assistant message rendering |

Host-provided globals stay external as configured (`react`, `mobx`, etc.).
`@kubernetes/client-node` and the agent SDK must only be imported from
`src/main/**` — they must not end up in the renderer bundle.

## File layout

The current `src/` only contains the template entry points; the
`gateway-api` layout described in AGENTS.md is the template's example and
is replaced by this structure (update the AGENTS.md Architecture section
as part of M0):

```text
src/
  common/
    bridge-store.ts        # ExtensionStore: { port, token } synced main -> renderer
    protocol.ts            # shared request/response and SSE event types
  main/
    index.ts               # onActivate: start bridge; onDeactivate: stop it
    claude/
      detect.ts            # locate Claude Code binary, read version
      session-manager.ts   # per-cluster session lifecycle over the SDK
    bridge/
      server.ts            # node:http server, routing, auth, CORS, SSE
    tools/
      kube-client.ts       # KubeConfig from catalog entity kubeconfig
      mcp-server.ts        # createSdkMcpServer with the three tools
      resources.ts         # kube_resources tool
      pod-logs.ts          # kube_pod_logs tool
      warning-events.ts    # kube_warning_events tool
  renderer/
    index.tsx              # clusterPages + clusterPageMenus registration
    api/
      bridge-client.ts     # fetch wrapper + SSE reader against the bridge
    components/
      chat-page.tsx        # page: onboarding gate or chat view
      chat-view.tsx        # message list + input + status strip
      markdown.tsx         # react-markdown wrapper (code blocks, links)
      onboarding.tsx       # Claude Code missing / not detected panel
    components/*.module.scss
```

## Main process

### Extension entry (`src/main/index.ts`)

`onActivate`: create the `BridgeStore`, run Claude Code detection, build
the HTTP server, store `{ port, token }` in the store. `onDeactivate`:
close the server and interrupt all live sessions. Errors during activation
must not throw out of `onActivate` — log via `Common.logger` and let
`GET /status` report the failure state.

### Claude Code detection (`src/main/claude/detect.ts`)

- Search order: `CLAUDE_CODE_PATH` env override, `claude` on `PATH`
  (`which`/`where`), then well-known locations (`~/.local/bin/claude`,
  npm/pnpm global bin dirs, `~/.claude/local/claude`).
- Validate by running `claude --version` (short timeout) and parsing the
  version string.
- Result: `{ found: boolean, path?, version?, error? }`. Do not attempt to
  verify login state in M0 — auth failures surface as a session error event
  with guidance to run `claude` in a terminal (the SDK reports them when
  the first message is sent).
- Detection runs at activation and can be re-triggered by `GET /status?refresh=1`
  so the onboarding panel has a "check again" button.

### Bridge server (`src/main/bridge/server.ts`)

- `node:http`, `listen(0, "127.0.0.1")` (ephemeral port).
- Per-launch token from `crypto.randomBytes(32).toString("hex")`; every
  request must carry `Authorization: Bearer <token>` or is answered `401`.
- CORS: reflect the request `Origin` only and set
  `Access-Control-Allow-Headers: Authorization, Content-Type`; answer
  `OPTIONS` preflights.
- Routes (M0 subset of the plan's D3 table):

| Route | Purpose |
| --- | --- |
| `GET /status` | detection result, bridge readiness |
| `GET /clusters/:id/events` | SSE stream of session events |
| `POST /clusters/:id/messages` | `{ text }` — send a user message |
| `POST /clusters/:id/interrupt` | stop the current turn |
| `DELETE /clusters/:id/session` | drop the session (new chat) |

- SSE: `Content-Type: text/event-stream`, heartbeat comment every 15 s,
  one JSON-encoded event per `data:` line with an `event:` type. On
  connect, replay the in-memory transcript so a remounted page restores
  the conversation within the app run.

Event types (define in `src/common/protocol.ts`, shared with the
renderer):

| Event | Payload |
| --- | --- |
| `status` | `{ state: "idle" \| "working" }` |
| `user_message` | `{ text }` (echo, for transcript replay) |
| `assistant_delta` | `{ text }` streaming chunk |
| `assistant_message` | `{ text }` finalized message |
| `tool_call` | `{ toolName, input }` |
| `tool_result` | `{ toolName, summary }` (truncated) |
| `turn_complete` | `{}` |
| `error` | `{ message, kind: "not_found" \| "auth" \| "other" }` |

### Session manager (`src/main/claude/session-manager.ts`)

One entry per cluster id: SDK `query()` with a streaming input generator
fed by `POST /messages`, plus the transcript buffer and SSE subscriber
list. SDK options:

- `pathToClaudeCodeExecutable`: from detection.
- `mcpServers`: the in-process server from `tools/mcp-server.ts`, bound to
  the cluster's kube client.
- `allowedTools`: exactly the three MCP tool names.
- `disallowedTools`: Claude Code built-ins (`Bash`, `Edit`, `Write`,
  `Read`, `Glob`, `Grep`, `WebFetch`, `WebSearch`, `NotebookEdit`, ...).
- `canUseTool`: allow the three MCP tools, deny everything else. No UI
  prompt exists in M0, so this callback must never block.
- `settingSources: []` — the user's global Claude config must not leak in.
- `cwd`: per-cluster scratch directory under
  `Main.LensExtension.getExtensionFileFolder()`.
- System prompt (append): the model is operating on Kubernetes cluster
  `<name>` through read-only tools; it cannot modify the cluster.

`interrupt()` maps to the SDK interrupt; `DELETE /session` disposes the
query and clears the transcript.

### Kubernetes access (`src/main/tools/kube-client.ts`)

Resolve the `KubernetesCluster` catalog entity by cluster id
(`Main.Catalog`), build a `KubeConfig` from `spec.kubeconfigPath` +
`spec.kubeconfigContext`. Cache one client set per cluster; surface
connection errors as tool results (text), never as thrown exceptions that
kill the session.

This is the main-process kubeconfig client — path 1 of the two viable tool
backends in D4. M0 deliberately uses it for all three read-only tools so
they do not depend on an open cluster frame. The alternative, bridging to
the renderer's `Renderer.K8sApi` to reuse Freelens' connection context, is
an equally legitimate design (not rejected) and is expected to back tools in
later milestones; it is out of scope for M0.

### Tools (`src/main/tools/`)

All read-only, names prefixed `kube_`:

1. `kube_resources` — list or get resources of any kind.
   Input: `{ apiVersion, kind, namespace?, name?, labelSelector?, limit? }`.
   Uses the discovery-based generic client (`KubernetesObjectApi`). Strips
   `metadata.managedFields`. Returns YAML; lists are truncated to `limit`
   (default 50) with a note when truncated.
2. `kube_pod_logs` — log snapshot.
   Input: `{ namespace, name, container?, tailLines? (default 200), grep? }`.
   `grep` is applied as a JS regex line filter after fetch. Output capped
   (e.g. 50 KiB) with a truncation note.
3. `kube_warning_events` — cluster health signal.
   Input: `{ namespace? }`. Lists `type=Warning` events (all namespaces
   when omitted), sorted by last timestamp, most recent first, capped.

Each tool returns `content: [{ type: "text", text }]`; errors return an
explanatory text result so the model can react.

## Common (`src/common/`)

- `bridge-store.ts`: `ExtensionStore` subclass holding `{ port, token }`,
  written by main, read by renderer. Follow the original AI extension's
  `PreferencesStore.aiProxyPort/aiProxyToken` pattern.
- `protocol.ts`: TypeScript types for routes, payloads, SSE events —
  imported by both processes, no runtime dependencies.

## Renderer

- `index.tsx`: register one `clusterPages` entry (id `chat`) and a
  `clusterPageMenus` entry titled **"Freelens for Claude"** with an SVG
  icon (add under `src/renderer/icons/`).
- `bridge-client.ts`: reads `{ port, token }` from the store. SSE must use
  `fetch()` with a `ReadableStream` reader, not `EventSource` —
  `EventSource` cannot send the `Authorization` header. Reconnect with
  backoff when the stream drops.
- `chat-page.tsx`: while `GET /status` reports Claude Code missing, render
  `onboarding.tsx` (what was checked, how to install, link to
  <https://code.claude.com>, "check again" button); otherwise `chat-view.tsx`.
- `chat-view.tsx`: transcript (user/assistant bubbles, in-progress
  assistant message from deltas, compact one-line notices for
  `tool_call`/`tool_result`), input (Enter sends, Shift+Enter newline,
  disabled while `working` — M0 has no message queueing), status strip
  (working spinner, Stop button wired to `/interrupt`, New chat wired to
  `DELETE /session`), error banner for `error` events with terminal
  guidance on `kind: "auth"`.
- `markdown.tsx`: `react-markdown` + `remark-gfm`; code blocks with a copy
  button; links open via `window.open` (external browser).
- MobX + `observer` from host globals for state; SCSS modules for styling,
  consistent with the template setup.

## Testing and validation

Unit tests (vitest, using the `test/freelens-extensions.ts` stub pattern —
extend the stub as needed):

- `detect.ts`: version parsing, search-order fallbacks (mock `execFile`).
- bridge auth: missing/wrong token gets `401`, `OPTIONS` preflight passes.
- protocol: SSE event encode/decode round-trip.
- tools: zod schema validation and `managedFields` stripping, `grep`
  filtering, truncation notes (mock kube client).

Validation gates (per AGENTS.md): `pnpm lint:fix && pnpm type:check &&
pnpm test:unit` and `pnpm build`. The existing integration test
(`integration/__tests__/extensions.tests.ts`) must stay green — the
extension must install and enable cleanly with no console errors even when
Claude Code is absent from the machine.
