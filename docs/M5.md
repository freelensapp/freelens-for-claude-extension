# M5 — UX overhaul

Specification for milestone M5, addressing issue
[#25](https://github.com/freelensapp/freelens-for-claude-extension/issues/25).
The reference for the target look and feel is the
[freelens-ai-extension](https://github.com/freelensapp/freelens-ai-extension)
chat layout shown in that issue.

The goal is purely presentational and interactional: move the chat from
"functional but rough" to a polished, theme-consistent UI. The bridge
architecture, the tool set, and the permission model are unchanged; the only
protocol change is persisting reasoning text (D3).

## Where issue #25 stands today

M0 through M4 delivered part of the issue's list. The plan below targets the
remaining gaps.

| Issue #25 request | Current state | M5 work |
| --- | --- | --- |
| "Write a message" prompt at the bottom | Done (M0): textarea pinned below the transcript | Reword placeholder; restyle as a composer (D1) |
| Model and mode widgets next to the prompt | Partially: selectors exist but sit in a status strip above the input | Move into the composer (D1) |
| Button to clear the session | Done: "New chat" button and `/clear` | Becomes an icon button in the composer (D1) |
| Token usage indicator | Done (M2): counter in the status strip | Move into the composer (D1) |
| Option to compact the session | Missing: only native auto-compaction | Compact action sending `/compact` (D4) |
| Auto-foldable reasoning sections | Partially (M3): live streaming only; reasoning is discarded when the turn completes | Persist reasoning per message and across replay (D3) |
| Better presentation of tool use | Partially (M2/M3): collapsible tool cards with nested subagent calls | Friendly titles, icons, status states (D5) |
| Better presentation of approvals | Largely done (M1/M2): foldable details, backup fold, collapses after resolution | Visual polish, copy action (D6) |

## Design

### D1. Composer

Replace the current three-row bottom area (status strip, shortcut chips,
input row) with a single rounded composer container plus the chips above it:

- **Textarea** at the top of the container: borderless, transparent
  background, placeholder "Write a message about this cluster...",
  auto-growing from 1 to about 6 lines. Enter sends, Shift+Enter inserts a
  newline, and the slash-command menu keeps its current key handling,
  anchored above the composer.
- **Control row** below the textarea, inside the container:
  - Left: icon buttons for New chat (clears the session, replacing the
    "New chat" text button), Compact (D4), and Available Tools (replacing
    the current `ToolsPanel` text button). Icons come from the host's
    `Renderer.Component.Icon` (Material icons), with tooltips.
  - Right: the token usage counter in the existing
    `in:N (cached:N) + out:N` format, the model selector, the mode selector
    (keeping the warning coloring for Accept all), and a round icon button
    that is Send when idle and Stop while a turn is in flight.
- The status strip is removed. Its remaining occupants move as follows:
  "Working..." is replaced by the Stop state of the send button plus a
  small animated "thinking" indicator at the end of the transcript;
  "conversation resumed" becomes a one-time system notice in the
  transcript.
- Quick-prompt chips keep their current behavior (shown when the input is
  empty and no turn is in flight) and render directly above the composer.

Files: `chat-view.tsx`, `chat-view.module.scss`, `tools-panel.tsx` (button
becomes an icon trigger; the popover itself is unchanged).

### D2. Transcript layout

Match the reference layout's visual hierarchy:

- **User messages** stay right-aligned accent-colored bubbles (the one
  element that must stand out).
- **Assistant messages** lose the gray bubble background and render as flat,
  full-width text blocks. The gray-on-gray bubble is the main reason the
  current UI reads as dimmed.
- **System notices** (compaction, resumed, legacy M1 replay lines) render as
  small centered text — the only place where dimmed styling is allowed.
- **Tool cards, approval cards, and local command output** stretch to the
  full transcript width instead of hugging the left edge at `max-width: 90%`.

Files: `chat-view.tsx`, `chat-view.module.scss`, `tool-card.module.scss`,
`permission-dialog.module.scss`.

### D3. Reasoning persistence (the one protocol change)

Today `assistant_thinking` deltas are live-only (`protocol.ts`), the
accumulated text is dropped when `assistant_message` arrives
(`chat-view.tsx` reducer), and nothing is replayed after a remount. M5 makes
reasoning a durable part of each assistant message:

- `SessionManager` accumulates thinking deltas per turn and attaches them to
  the next emitted `assistant_message`, whose payload becomes
  `{ text, reasoning? }`; the accumulator resets after each message.
  Because `assistant_message` is already in the persisted-event set, replay
  and disk persistence come for free.
- The live `assistant_thinking` event is unchanged and still drives the
  streaming fold.
- The renderer's `assistant` chat item gains `reasoning?: string` and
  renders the existing `ReasoningFold` above the markdown body, collapsed by
  default for completed messages. The live auto-open/auto-fold behavior is
  unchanged.
- Old transcripts without the field render exactly as today.

Files: `protocol.ts`, `session-manager.ts`, `chat-view.tsx`, plus a
session-manager unit test for accumulate/attach/reset.

### D4. Manual compaction

Claude Code already handles `/compact` natively: the session manager emits
`local_command_output` for its printable output and a `compaction` event at
the boundary, and both render today. M5 only adds an affordance:

- A Compact icon button in the composer (D1) sends the literal `/compact`
  message through the existing message route. Disabled while a turn is in
  flight or before any usage has accrued.
- The compaction notice distinguishes the trigger: "Conversation compacted
  manually" versus "... automatically to save context" (the `trigger` field
  already exists on the event).

No bridge or protocol changes. Files: `chat-view.tsx`.

### D5. Tool call presentation

Keep the collapsible card model, improve scanability:

- A static map from tool names to a human title and Material icon:
  for example `freelens_resources` becomes "Read resources",
  `freelens_pod_logs` "Read pod logs", `freelens_kubectl` "Run kubectl",
  `Agent` "Subagent", and external MCP tools fall back to
  `server: tool` with a plug icon. The raw tool name stays visible in the
  expanded body.
- A status affordance on the collapsed header: the existing pulsing dot
  while running, a subtle check when finished, and error coloring when the
  result summary starts with `Error`.
- Legacy replayed M1 events (`tool_call`/`tool_result` without `callId`)
  restyle as system notices (D2) — no functional change.

Files: `tool-card.tsx`, `tool-card.module.scss`, `chat-view.tsx`.

### D6. Approval card polish

The card already covers the issue's functional asks (foldable action
details, backup of the resource to be changed, one-line fold after
resolution). Remaining polish:

- Header with a warning icon next to the action title, visually distinct
  from ordinary tool cards (warning-tinted border or background).
- Copy button on the YAML/diff blocks, reusing the markdown code-block
  pattern.
- Approve styled as the primary action and Deny as the destructive one,
  instead of two equal-weight buttons.

Files: `permission-dialog.tsx`, `permission-dialog.module.scss`.

### D7. Theming and typography

- Inline code (backticks) gets chip styling: a subtle background, small
  padding and radius, and **full-contrast text color** — never dimmed.
- Reduced opacity is reserved for system notices and card metadata; message
  content, code, and controls render at full text color.
- Replace hardcoded colors (`#3d90ce`, the `rgba(127, 127, 127, ...)`
  family) with host theme variables and fallbacks (`--primary`,
  `--textColorPrimary`, `--borderColor`, `--mainBackground`,
  `--colorError`, ...), auditing against the actual variable set the
  Freelens host exposes so both light and dark themes work.

Files: all `*.module.scss` under `src/renderer/components/`,
`markdown.module.scss` in particular.

### D8. Scrolling

- Audit the height chain from the cluster page container down to
  `.chatView` so the transcript is always the single scroll container and
  the composer never scrolls out of view, regardless of window size.
- Replace the unconditional scroll-to-bottom effect with stick-to-bottom:
  autoscroll only when the user is already within a small threshold of the
  bottom; otherwise leave the viewport alone and show a floating
  "jump to latest" button.

Files: `chat-view.tsx`, `chat-view.module.scss`, `chat-page.module.scss`.

## Implementation slices

Each slice is an independently shippable PR-sized change, ordered so that
later slices build on earlier ones:

1. **Theming and typography** (D7): SCSS-only; inline-code chips, dimming
   audit, theme variables.
2. **Transcript layout and scrolling** (D2, D8): flat assistant messages,
   system notices, full-width cards, stick-to-bottom.
3. **Composer** (D1, D4): the structural rewrite of the bottom area plus
   the Compact button.
4. **Reasoning persistence** (D3): the protocol/session-manager change and
   the per-message fold, with unit tests.
5. **Tool and approval card polish** (D5, D6): titles, icons, status
   states, copy actions.

Per the process note in [PLAN.md](./PLAN.md), implementation runs on pull
requests; each slice follows the usual validation
(`pnpm lint:fix && pnpm type:check && pnpm test:unit`, plus `pnpm build`).

## Acceptance criteria

Mirroring issue #25:

- The prompt sits at the bottom in a composer with a "Write a message"
  style placeholder.
- Model and mode selectors sit next to the prompt, inside the composer.
- A clear-session button is available in the composer.
- The session token usage is visible in the composer and a Compact action
  triggers `/compact` with visible feedback.
- Reasoning renders as a fold on every assistant message: auto-open while
  streaming without answer text, folded once the answer arrives, and
  preserved (collapsed) on completed and replayed messages.
- Tool use renders as titled, iconed, collapsible cards with visible
  running/finished state; system messages are visually distinct from
  conversation content.
- Approvals show the action title with foldable details, including the
  backup of the resource to be changed, and fold to one line after
  resolution.
- Inline code renders at full contrast; dimmed text appears only on system
  messages.
- The conversation is reliably scrollable with stick-to-bottom behavior and
  a jump-to-latest affordance.
- User prompts are clearly separated from assistant responses in both
  light and dark themes.

## Out of scope

- Monaco or any heavyweight editor/syntax highlighter (bundle size).
- Cost display (dropped by design, PLAN.md D8).
- Editing the proposed input in the approval card (post-1.0 backlog).
- Message-level actions (edit, regenerate, per-message copy of whole
  messages) beyond what exists today.
- Any change to tools, permissions, session lifecycle, or the bridge API
  other than the `assistant_message.reasoning` field (D3).
