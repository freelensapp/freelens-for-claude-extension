# M4 — Documentation and Release Readiness: Implementation Specification

Status: **delivered** — merged in
[#24](https://github.com/freelensapp/freelens-for-claude-extension/pull/24)
(approved plan: [PLAN.md](./PLAN.md), issue
[#7](https://github.com/freelensapp/freelens-for-claude-extension/issues/7);
M0 through M3 delivered and confirmed working by the maintainer).

Success criterion: a maintainer can cut the first public release using only
the existing automation and the documented process — the README fully
documents installation, requirements, every shipped feature, the security
model, and troubleshooting; the package metadata describes the extension
accurately and its minimum Freelens version claim is verified; a release
document walks through the automated version-bump, tag, and publish chain
and records the pre-release checklist, including the D2 contact with
Anthropic before any announcement. All validation gates stay green.

## Scope

In scope:

1. README rewrite: replace the extension-template stub with complete user
   documentation of the shipped extension.
2. Package metadata polish: an accurate `description`, richer `keywords`,
   a `homepage` field, and a verified `engines.freelens` floor.
3. Release process document (`docs/RELEASE.md`): how a release is cut with
   the already-present GitHub Actions automation, plus the pre-release
   checklist.

Out of scope (recorded as the post-1.0 backlog in
[PLAN.md](./PLAN.md#post-10-backlog); none of it blocks a first release):

- Per-cluster "always allow" tool lists.
- User-defined subagents (a preferences surface over the SDK `agents`
  option; only the built-in `cluster-analyzer` ships).
- Editing the proposed input in the approval card, and MCP tool search.
- A visible-Freelens-terminal variant of the kubectl/helm tools.
- Hooks passthrough (SDK `hooks` stays unused; `PreToolUse` denials would
  bypass `canUseTool`, which is why the broker must remain the only gate).
- Bridging tool execution to the renderer's `Renderer.K8sApi` (PLAN D4
  path 2) — retained as a design option, not needed by any shipped tool.
- Screenshots in the README — they need a running Freelens with a live
  cluster and an authenticated Claude Code, which CI does not have; the
  README structure leaves a marked place for the maintainer to add them.
- The marketplace/announcement step itself — a maintainer action gated on
  the D2 legal contact, tracked in the release checklist, not in code.

## Research notes (current state)

Findings that shape this spec, verified in the repository at M3 merge
(`a2b4636`):

- **README.md is still the template stub**: overview says only "Freelens
  extension for Claude" and links the generic wiki page about creating
  extensions. The build-from-source, install-tarball, and static-check
  sections are accurate and worth keeping nearly as-is.
- **Release automation is already complete** and needs no M4 changes
  (also: workflow files are not editable via the Claude GitHub App, so any
  change there would have to be maintainer-applied — none is required):
  - `npm-version.yaml` — manual `workflow_dispatch` with a `newversion`
    input (exact `X.Y.Z` or a semver keyword); bumps `package.json` via
    `pnpm bump-version`, opens/updates the `automated/npm-version` PR
    titled `Automated npm version vX.Y.Z`. Also auto-runs with
    `prerelease` after each published release.
  - `tag.yaml` — when that PR merges (or `/tag` is commented on its closed
    issue), creates the `vX.Y.Z` tag from `main`, only for stable
    `X.Y.Z` versions (prereleases never tag).
  - `release.yaml` — on tag push: guards that the tag matches
    `package.json` version, builds, packs, publishes to npm (provenance or
    `NPM_TOKEN`; dist-tag `next` for hyphenated prerelease versions,
    `latest` otherwise), generates an SPDX SBOM, sha256 checksums, and
    creates the GitHub release with the `.tgz` and checksum assets.
- **Integration tests** (`integration-tests.yaml`) build the app from
  `freelensapp/freelens` (matrix: v1.10.0), install the packed `.tgz` at
  runtime, and fail on renderer/main error output — this is the automated
  install smoke test, per PR and on `main`.
- **package.json**: `version` 0.1.0 (no tags or releases exist yet),
  `description` "Freelens extension for Claude" (the template
  placeholder), `engines.freelens` `^1.8.0` — inherited from the template
  and never verified. The extension is developed against
  `@freelensapp/extensions` 1.10.x, its M3 research was validated against
  the 1.10.2 declarations, and CI integration-tests only Freelens 1.10.0.
- **AGENTS.md constraints that apply to the new documents**: no emoji
  anywhere, Markdown is formatted by trunk (`pnpm trunk:fix`), and
  `pnpm knip:check` runs over `package.json` after a build.

## README specification

Replace `README.md` with the structure below. Keep the existing badges,
the markdownlint disable/enable comments around them, and the license
footer. Keep the build-from-source content (prerequisites, build, pack,
install tarball, static checks) as a late section. All feature claims must
match the shipped code — when in doubt, the tool names, preference fields,
and behaviors in `src/` and the M0-M3 specs are the source of truth, and
the M0-M3 specs should be linked for detail rather than duplicated.

1. **Title and badges** — unchanged.
2. **Overview** — what the extension is: a Claude-powered chat embedded in
   Freelens, one conversation per cluster, where Claude acts on the
   cluster only through typed, permission-gated tools. Name the deliberate
   difference from
   [freelens-ai-extension](https://github.com/freelensapp/freelens-ai-extension):
   this extension is a frontend to the **user's own Claude Code
   installation** — no provider selection, no API keys stored, billing is
   whatever the user's Claude Code uses (subscription or API). Link
   `docs/PLAN.md` for the design decisions. State plainly: this is a
   community project, not an official Anthropic product; "for Claude"
   is nominative use of the trademark.
3. **Requirements** — Freelens (the verified `engines` floor; note that CI
   tests against 1.10.0), Claude Code installed and logged in. Note that
   every Claude Code authentication method is inherited unchanged
   (subscription login, `ANTHROPIC_API_KEY`, Bedrock, Vertex) — the
   extension never sees credentials. Node.js is required only for building
   from source.
4. **Installation** — from the Freelens Extensions page by npm name
   (`@freelensapp/for-claude-extension`) once published, or by downloading
   the `.tgz` from GitHub releases and dragging it into the Freelens
   window. Cross-link the build-from-source section.
5. **Getting started** — open a cluster, choose "Freelens for Claude" in
   the cluster side menu; the onboarding panel appears when Claude Code is
   not detected (with the path-override preference as the fix); first
   prompt via a shortcut chip (for example "Cluster health") or free text;
   right-click a Pod or Event and choose "Ask Claude". Placeholder comment
   for a maintainer-supplied screenshot.
6. **Features** — subsections, each a short paragraph or list:
   - **Chat**: streamed markdown answers with code blocks, a live
     "Reasoning" fold, stop and retry, per-cluster transcripts that
     survive restarts (Claude Code session resume), a token-usage counter,
     automatic context compaction (with a transcript notice), and a model
     picker (Claude Code aliases).
   - **Built-in tools**: a table of the twelve `freelens_` tools plus
     `freelens_kubectl` and `freelens_helm` — name, read-only or mutating,
     one-line purpose. Note kubectl/helm receive argv only (no shell) and
     are always pinned to the current cluster.
   - **Permissions and safety**: the three per-cluster modes (read-only /
     approve, the default / accept-all, never persisted); the approval
     card with proposed YAML, current-resource backup, and diff; the
     pod-logs consent preference; Claude Code's own shell and filesystem
     tools are disabled — the cluster is reachable only through the typed
     tools above.
   - **Slash commands and shortcuts**: `/` autocomplete over Claude Code's
     commands, local command output in the transcript, `/clear` mapping to
     New chat; built-in and user-defined prompt chips.
   - **Cluster analyzer subagent**: read-only delegation for broad
     investigations, nested tool calls visible and still permission-gated;
     toggleable in preferences.
   - **User MCP servers**: opt-in, Claude-Desktop-style JSON, additive to
     the built-in tools, every external call requires approval; the
     Available Tools panel lists everything live.
7. **Preferences reference** — a table of every setting under the
   extension's preferences page with its default and one-line meaning:
   pod-logs approval requirement (on), pod-logs tail-lines default, custom
   agent rules, Claude Code path override, default model, MCP enable
   (off) + JSON configuration, analysis subagent (on), prompt shortcuts
   JSON. Note which settings apply at the next new chat.
8. **Security model** — the bridge is `127.0.0.1`-only with a per-launch
   bearer token; the extension stores no credentials and never talks to
   Anthropic itself; `settingSources: []` and `strictMcpConfig` keep the
   user's global Claude Code configuration out of cluster chats; the
   permission broker is the single gate for every mutating, consent-gated,
   and external tool call; kubectl/helm argv is validated
   (`--kubeconfig`/`--context`/`--kube-context` rejected, cluster
   targeting injected by the extension).
9. **Privacy** — whatever the tools return (resource YAML, logs, events)
   becomes part of the conversation sent to Anthropic through the user's
   own Claude Code session, under the user's own plan and Anthropic's
   terms; treat the chat like pasting `kubectl` output into Claude.
10. **Troubleshooting** — Claude Code not detected (PATH, the override
    preference); not logged in (run `claude` in a terminal); where
    per-cluster scratch directories and transcripts live (under the
    extension's data path); main-process errors appear in the terminal
    that launched Freelens, renderer errors in DevTools.
11. **Build from the source** — the existing content, updated only where
    it references the wiki stub.
12. **License** — unchanged.

## Release process document (`docs/RELEASE.md`)

A short document with two parts:

1. **Mechanics** (describing the existing automation, no new workflows):
   - Run the "Automated npm version" workflow (`workflow_dispatch`) with
     the target version (exact `X.Y.Z` for a release; the default
     `prerelease` is for dev packs).
   - Review and merge the `Automated npm version vX.Y.Z` PR; on merge the
     tag workflow creates `vX.Y.Z` from `main` (stable versions only).
   - The tag push triggers the release workflow: version-match guard,
     build, pack, npm publish (dist-tag `latest`; prereleases go to
     `next`), SBOM + sha256 checksums, GitHub release with the `.tgz`
     assets.
   - After the GitHub release is published, the version workflow auto-runs
     once more with `prerelease` to reopen the dev version cycle.
2. **Pre-release checklist**:
   - `pnpm lint:check && pnpm type:check && pnpm test:unit && pnpm build`
     green on `main`; integration tests green.
   - Manual smoke test in a real Freelens: onboarding with Claude Code
     absent, a read-only question, one approved mutation, one denial,
     restart-and-resume.
   - README versions/claims still accurate (Freelens floor, feature list).
   - **D2 gate**: before any announcement or marketplace submission,
     contact Anthropic to confirm the terms-of-service posture recorded in
     [PLAN.md D2](./PLAN.md) — publishing the GitHub release itself is
     fine (the project has been public throughout); announcing is what the
     maintainer explicitly deferred until after legal review.
