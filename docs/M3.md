# M3 ‚Äî Beyond Parity: Implementation Specification

Status: **ready for implementation** (approved plan: [PLAN.md](./PLAN.md),
issue [#7](https://github.com/freelensapp/freelens-for-claude-extension/issues/7);
M0, M1 and M2 delivered and confirmed working by the maintainer).

Success criterion: the user pastes an MCP server into preferences and Claude
uses its tools after an approval card names the server and shows the full
input; typing `/` in the chat input pops up the available commands and
`/compact` runs with its output in the transcript; clicking the "Cluster
health" shortcut chip sends a canned prompt; on a broad "find everything
wrong in this cluster" question the agent delegates to the read-only
`cluster-analyzer` subagent, whose tool calls render indented under the
delegation card; and an approved `kube_kubectl` call runs visibly in the
Freelens terminal, with the captured output returned to the model.

## Scope

In scope (the M3 milestone from PLAN.md, plus the two panels M2 explicitly
deferred here):

1. User MCP-server passthrough: a preferences toggle plus Claude-Desktop-style
   JSON configuration, passed to the Agent SDK alongside the built-in
   `freelens-kube` server; every external tool call goes through the approval
   card.
2. Claude Code subagents: a built-in read-only `cluster-analyzer` subagent
   the main agent can delegate deep investigations to, keeping large tool
   outputs out of the main context; nested tool calls stay visible and stay
   gated by the same permission broker.
3. Slash commands: input starting with `/` is forwarded to Claude Code, an
   autocomplete popup lists the available commands, and local command output
   renders in the transcript.
4. Prompt shortcuts: built-in and user-defined quick prompts rendered as
   chips above the input.
5. Kubectl tool (`kube_kubectl`): after approval, the command runs in
   Freelens' own terminal ‚Äî visible to the user, authenticated by the
   terminal's cluster kubeconfig ‚Äî with best-effort output capture returned
   to the model.
6. Available Tools panel: an informational popover listing the built-in
   `kube_` tools and the connected MCP servers with their tools (deferred
   from M2).
7. Reasoning fold: live thinking deltas render in a collapsed "Reasoning"
   fold above the streaming answer (deferred from M2).

Out of scope (consciously not in M3):

- Per-cluster "always allow" tool lists ‚Äî still deferred (the original has
  none; M2 already dropped it from the parity backlog).
- User-defined subagents ‚Äî M3 ships only the built-in `cluster-analyzer`
  definition; a preferences surface for custom agents can follow once the
  rendering and gating machinery has been proven.
- Hooks passthrough ‚Äî the SDK `hooks` option stays unused; note the SDK
  caveat that `PreToolUse` denials bypass `canUseTool`, which is exactly why
  the broker must remain the only gate.
- Editing the input in the approval card, MCP tool search, and Windows
  output capture for the Kubectl tool (visible execution still works; see
  the Kubectl section).

## Parity and SDK research notes

Findings from `freelensapp/freelens-ai-extension@main` and the installed
`@anthropic-ai/claude-agent-sdk` 0.3.204 that shape this spec:

- **The original's MCP support** (`preferences-store.ts`, `mcp-agent.ts`):
  an `mcpEnabled` switch plus one raw JSON textarea (`mcpConfiguration`,
  Claude-Desktop-style `{"mcpServers": {...}}`, commit-on-blur), wired
  through LangChain's `MultiServerMCPClient`. Enabling MCP **replaces** the
  built-in kube tools entirely (either/or agents), tool namespacing is
  disabled, and every MCP call interrupts with only the tool *name* shown.
  M3 keeps the storage format (switch + JSON textarea) but deliberately
  improves on all three weaknesses: user servers are **additive** to the
  built-in tools (the SDK namespaces per server, so nothing collides),
  names stay qualified, and the approval card shows the full input as YAML
  like any mutating tool.
- **Slash commands / prompt shortcuts do not exist in the original** ‚Äî its
  input bar has only send, clear, a tools toggle, and the bypass switch.
  Both features here are beyond parity, promised by PLAN M3.
- **The original's terminal integration is human-only**: shell code blocks
  get a Play button that opens a "FreeLens AI" terminal tab
  (`createTerminalTab` + `terminalStore.sendCommand`); the agent itself is
  told "there is NO shell". M3's `kube_kubectl` reuses the same Freelens
  APIs but is agent-initiated behind an approval.
- **The original's sub-agents** (supervisor / analyzer / operator /
  conclusions) are LangGraph nodes that exist because it had to build the
  agent loop itself. Claude Code already owns the loop, so M3 maps the one
  genuinely useful idea ‚Äî a read-only analyzer with its own context ‚Äî onto
  a Claude Code subagent.
- **Available Tools panel** (`available-tools.tsx`): a scrollable list in
  the input area behind a toggle button, fed by a static descriptor array
  (or live MCP discovery in MCP mode). M3 matches, fed by a bridge route.
- **Reasoning fold** (`message.tsx`): a native `details` element with a
  "Reasoning" summary that auto-opens while reasoning is the only content
  streaming and folds once answer text arrives. M3 matches the rendering;
  the content is live-only (see the renderer section).

Relevant SDK surface (verified against the installed 0.3.204 types):

- `Options.mcpServers: Record<string, McpServerConfig>` accepts stdio
  (`command`, `args?`, `env?`), `sse` and `http` (`url`, `headers?`)
  configs alongside in-process `sdk` servers; external tools are named
  `mcp__<server>__<tool>`.
- `Options.strictMcpConfig: boolean` ignores MCP config from settings
  files, plugins, and agent frontmatter ‚Äî only the `mcpServers` option
  counts.
- `Options.agents: Record<string, AgentDefinition>` defines subagents
  programmatically (`description`, `prompt`, `tools?`, `model?`);
  programmatic agents work under `settingSources: []`, which only controls
  filesystem settings. Delegation surfaces as a `tool_use` block for the
  `Agent` tool (the SDK's rename of `Task`).
- Subagent activity: `SDKAssistantMessage`/`SDKUserMessage` carry
  `parent_tool_use_id` (non-null inside a subagent). By default only
  tool_use/tool_result blocks are forwarded ‚Äî subagent prose stays out of
  the stream, which is exactly what M3 wants (the summary comes back as
  the `Agent` tool result). `canUseTool` **does fire for tool calls inside
  subagents** (its options carry `agentID`), so broker gating holds.
- Slash commands: the `system`/`init` message lists `slash_commands:
  string[]` and `mcp_servers: { name, status }[]`;
  `SDKLocalCommandOutputMessage` (`subtype: "local_command_output"`)
  carries printable command output; `/compact` surfaces as the
  `compact_boundary` M2 already renders; `/clear` emits a
  `conversation_reset` message.
- `Query.mcpServerStatus()` returns per-server status
  (`connected | failed | needs-auth | pending | disabled`) and the
  discovered tools with descriptions ‚Äî the data source for the Available
  Tools panel.

Freelens terminal facts (verified against `@freelensapp/core` 1.10.2
declarations and bundle):

- Terminal control is **renderer-only**: `Renderer.Component.createTerminalTab({ title, id? })`
  and `Renderer.Component.terminalStore.sendCommand(command, { enter, tabId })`.
  Both act on the dock of the cluster frame the code runs in ‚Äî which is the
  chat page's frame, so targeting is automatic. There is no main-process
  terminal API, so the Kubectl tool must round-trip through the renderer.
- `sendCommand` resolves when keystrokes are delivered, not when the
  command finishes; output capture means subscribing to
  `terminalStore.getTerminalApi(tabId).on("data", ...)`, which yields the
  raw PTY stream (ANSI escapes, echo, prompt included). Completion needs a
  sentinel.
- Inside a Freelens terminal, `KUBECONFIG` already points at Freelens'
  per-cluster proxy kubeconfig, so plain `kubectl` is authenticated against
  the right cluster with no extra flags. The kubectl binary is the user's
  configured path (`Common.App.Preferences.getKubectlPath()`) or the
  bundled one on the terminal's `PATH`.

## Protocol additions (`src/common/protocol.ts`)

All additions are optional fields or new event types; M2 transcripts replay
unchanged.

New SSE event types in `SessionEventMap`:

| Event | Payload | Persisted |
| --- | --- | --- |
| `assistant_thinking` | `{ delta: string }` | no (live-only, like `assistant_delta`) |
| `local_command_output` | `{ content: string }` | yes |
| `terminal_run` | `{ runId: string, command: string, capture: boolean }` | no |

Changed events:

| Event | Change |
| --- | --- |
| `tool_call` | + `parentCallId?: string` (set when the call happened inside a subagent: the `Agent` call's id) |
| `tool_result` | + `parentCallId?: string` |
| `session_meta` | + `slashCommands?: string[]` (from `init`), + `mcpServers?: { name: string; status: string }[]` (external servers only) |

New request/response types and routes:

| Route | Purpose |
| --- | --- |
| `GET /clusters/:id/tools` | Available Tools panel data: `{ builtin: { name, description, mutating }[], mcp: { name, status, tools: { name, description? }[] }[] }` |
| `POST /clusters/:id/terminal-result` | `{ runId: string, output?: string, exitCode?: number, error?: string }` ‚Äî the renderer reporting a `terminal_run` outcome |

`RESERVED_MCP_SERVER_NAME = "freelens-kube"` is exported so the config
parser and tests share it.

## Main process

### User MCP-server passthrough

- `mcp-config.ts` (new): `parseUserMcpConfig(json: string)` returns
  `{ servers: Record<string, McpServerConfig>, errors: string[] }`. Rules:
  - top level must be an object with an `mcpServers` object (the
    Claude-Desktop format the original uses);
  - each entry must match one of the SDK shapes ‚Äî stdio (`command` string,
    optional `args` string array, optional `env` string record), `sse` or
    `http` (`url` string, optional `headers` record); unknown fields are
    dropped, unknown `type` values rejected;
  - the reserved name `freelens-kube` is rejected;
  - malformed JSON or a malformed entry produces an entry in `errors` and
    is skipped ‚Äî never a throw.
- `startQuery` merges the parsed servers into the `mcpServers` record when
  the `mcpEnabled` preference is on, sets `strictMcpConfig: true`
  unconditionally (belt to `settingSources: []`'s braces), and emits one
  non-fatal `error` event listing any parse errors ("MCP configuration
  ignored: ..."). Like custom agent rules, config changes apply from the
  next session start; the preferences hint says so.
- After `init`, the external entries of `mcp_servers` (everything except
  `freelens-kube`) are kept and included in subsequent `session_meta`
  events as `mcpServers`; a server whose status is not `connected`
  additionally emits a non-fatal `error` event naming it.
- `canUseTool` gating for external tools: a tool name starting with
  `mcp__` that does not belong to `freelens-kube` routes through
  `decideMutating` with a generic descriptor ‚Äî external tools cannot be
  classified, so they are treated as mutating: `readOnly` mode denies
  them, `approve` prompts on every call (parity with the original's
  every-call interrupt, but with the full input visible), `acceptAll`
  auto-approves. `describeApproval` gains the fallback case: action title
  `USE MCP TOOL`, subtitle `<server> / <tool>`, `proposedValue` = the tool
  input. Unknown non-MCP tool names remain denied.

### Subagents (`subagents.ts`)

- One built-in definition, `cluster-analyzer`:
  - `description`: "Read-only deep investigation of cluster state:
    correlating resources, warning events and pod logs across many
    objects. Use for broad health checks or root-cause hunts that would
    need many tool calls, and return a concise findings report."
  - `prompt`: a read-only investigator system prompt ‚Äî investigate using
    the available kube tools, never suggest mutations it cannot perform,
    end with a compact findings summary (the only text that returns to the
    main agent).
  - `tools`: the qualified read-only tool names (including
    `mcp__freelens-kube__kube_pod_logs` ‚Äî the consent gate still applies
    through `canUseTool`, which fires inside subagents).
  - No `model` (inherit) and no other options.
- `startQuery` passes `agents: { "cluster-analyzer": ... }` when the
  `subagentsEnabled` preference is on (default on), and removes `Task`
  from `DISALLOWED_BUILTIN_TOOLS` in that case, adding `Agent` to the
  `allowedTools` list instead ‚Äî delegation itself is safe; every tool the
  subagent calls is still individually gated. With the preference off,
  `Task` stays disallowed exactly as today. (`Task` is the legacy alias
  the SDK maps to `Agent`; keep both out when disabled.)
- `handleMessage`: `assistant` and `user` messages with a non-null
  `parent_tool_use_id` map their tool_use/tool_result blocks to
  `tool_call`/`tool_result` events carrying `parentCallId`. Subagent text
  blocks are not forwarded by the SDK by default and M3 keeps it that way
  (`forwardSubagentText` unset). The `Agent` call itself flows through the
  existing tool_call/tool_result path (its input carries `description` and
  `prompt`; its result is the subagent's report), so the M2 card
  machinery renders delegation for free.

### Slash commands

- No main-process parsing: the raw user text (including a leading `/`) is
  pushed to the SDK as today; Claude Code processes commands natively.
- `handleMessage` additions: `system`/`init` captures `slash_commands`
  into the session meta (emitted as `session_meta.slashCommands`); a
  message with `subtype: "local_command_output"` emits the new
  `local_command_output` event (persisted); a `conversation_reset`
  message emits a `notice`-style `local_command_output` ("Conversation
  cleared by /clear") ‚Äî the renderer additionally intercepts `/clear`
  before sending (see below), so this is a fallback for commands typed in
  odd ways.

### Kubectl tool (`kubectl.ts`)

Contract (registered in `mcp-server.ts`; **mutating** name set ‚Äî it can do
anything kubectl can):

```text
kube_kubectl
  args: string[]        # kubectl argv, WITHOUT the leading "kubectl"
  timeoutSeconds?: number  # default 120, max 600
```

- Validation in the tool: `args` must be non-empty; every element must be
  free of NUL and newline characters. The display command is `kubectl`
  plus the args, each shell-quoted (single-quote wrapping with `'\''`
  escaping) ‚Äî the model provides argv, never a shell string, so pipes,
  redirects and chaining are structurally impossible.
- Approval: routed through `decideMutating` like other mutating tools.
  `describeApproval` case: action title `RUN KUBECTL`, `proposedValue` =
  the full quoted command line. `readOnly` denies, `approve` prompts,
  `acceptAll` auto-approves.
- Execution round-trip (main cannot touch the terminal):
  1. after approval the tool handler generates a `runId`, registers a
     pending entry (promise + timeout), and emits
     `terminal_run { runId, command, capture }` on the cluster's SSE
     stream; `capture` is false when the platform is Windows (PowerShell
     sentinel syntax differs; visible execution still works).
  2. the renderer executes it (see below) and reports via
     `POST /clusters/:id/terminal-result`.
  3. the handler resolves with the captured output (truncated through the
     existing `kube-format` byte cap) plus the exit code, or ‚Äî on
     `capture: false`, capture failure, or timeout ‚Äî a text result stating
     the command was sent to the Freelens terminal but output could not be
     captured, so the model asks the user what they saw instead of
     guessing.
  - If no SSE subscriber is connected for the cluster (chat page closed),
    the tool fails fast with "the Freelens chat page must be open to run
    terminal commands". The session manager exposes a `hasSubscribers`
    check for this.
- On `interrupt`, new chat, and dispose, pending terminal runs are
  rejected (same lifecycle as pending permissions).

### Bridge routes (`server.ts`)

- `GET /clusters/:id/tools`: static built-in descriptors (name, first
  sentence of the tool description, mutating flag ‚Äî exported from
  `mcp-server.ts` next to the name sets) plus, when a query is live,
  `Query.mcpServerStatus()` filtered to external servers (name, status,
  tool names/descriptions). No live query: `mcp: []`.
- `POST /clusters/:id/terminal-result`: validates the body shape and hands
  it to the session manager's pending-run table; `404` for an unknown
  `runId`, `200` otherwise.

## Renderer

### Slash-command popup (`slash-menu.tsx`)

- When the input starts with `/` and contains no whitespace yet, a popup
  above the input lists matching commands from
  `session_meta.slashCommands`, filtered by the typed prefix, keyboard
  navigable (up/down/tab/enter to complete, escape to dismiss). Empty or
  absent `slashCommands` (e.g. replayed session before the first turn)
  shows nothing ‚Äî typing a command still works, completion is a
  convenience.
- Sending is unchanged: the text goes through the normal `send` path and
  renders as a user bubble; command output arrives as
  `local_command_output` and renders as a monospace transcript item
  (styled like the tool notices).
- One interception: a trimmed input of exactly `/clear` triggers the
  existing New chat action instead of being sent ‚Äî our transcript and
  session id must reset together, and New chat is already that mechanism.

### Prompt shortcut chips

- Rendered above the input row when the input is empty and no turn is in
  flight. Built-in chips: "Cluster health" ("Give me an overall health
  check of this cluster: nodes, failing workloads, recent warning
  events."), "Failing pods" ("Find pods that are failing or restarting and
  explain why."), "Recent warnings" ("Summarize recent warning events and
  what they mean."). User chips parsed from the `promptShortcuts`
  preference (title + prompt strings only; entries missing either are
  skipped). Clicking a chip sends the prompt through the normal `send`
  path.

### Reasoning fold

- The reducer accumulates `assistant_thinking` deltas into the current
  streaming item's `reasoning` string; the assistant bubble renders a
  `details` fold ("Reasoning") above the answer text when `reasoning` is
  non-empty ‚Äî auto-open while reasoning is the only content, folded once
  answer text arrives, manual toggles respected afterwards (the original's
  behavior). Deliberate deviation from the original: reasoning is
  **live-only** ‚Äî `assistant_thinking` is not persisted, so the fold
  disappears from replayed transcripts (thinking is ephemeral by nature
  and Claude Code does not return it on resume).
- Main side: `stream_event` handling adds the `thinking_delta` case,
  emitting `assistant_thinking`.

### Nested tool cards

- Tool events carrying `parentCallId` render indented under the matching
  `Agent` delegation card (which the M2 card machinery already renders
  from its `tool_call`/`tool_result` pair; its header shows the
  `description` from the input and a "subagent" tag instead of a kube
  argument summary). Orphaned parented events (no matching card in the
  transcript) fall back to top-level rendering.

### Terminal runner (`api/terminal-runner.ts`)

Handling for `terminal_run` events (wired in `chat-view.tsx`, which owns
the SSE subscription ‚Äî the chat page lives in the cluster frame, which is
exactly where the terminal APIs act):

1. Reuse-or-create a dock tab titled "Freelens for Claude" with a fixed id
   (the original's pattern) via `createTerminalTab`.
2. `capture: true`: subscribe to `terminalStore.getTerminalApi(tabId)`'s
   `data` events, then `sendCommand` the command wrapped as
   `printf '<BEGIN sentinel>\n'; <command>; printf '\n<EXIT sentinel %s>\n' "$?"`
   with `enter: true`. Accumulate data between the **echoed output** of
   the BEGIN sentinel and the EXIT sentinel (matching the printf *output*
   lines, not the echoed command line, which also contains the sentinel
   text ‚Äî match on line start), strip ANSI escape sequences with a regex,
   parse the exit code from the EXIT sentinel, and post
   `{ runId, output, exitCode }`. A 5 KB rolling cap guards the
   accumulator; the main process truncates further.
3. `capture: false`: `sendCommand` the bare command and immediately post
   `{ runId, error: "output capture not supported on this platform" }` ‚Äî
   the command still runs visibly.
4. Failures (terminal API never ready, sentinel not seen before the
   deadline sent in the event payload) post `{ runId, error }`. Every path
   posts exactly once.

This is explicitly best-effort: PTY scraping has no protocol. The tool's
contract (above) makes the model degrade gracefully when capture fails,
and the user always sees the real execution in the terminal.

### Available Tools panel (`tools-panel.tsx`)

- A toggle button in the status strip ("Tools") opens a popover listing:
  built-in tools grouped as read-only / mutating (name + description from
  `GET /clusters/:id/tools`), then each external MCP server with its
  status and tool list. Fetched on open, not cached. Purely informational
  ‚Äî no enable/disable (parity: the original's panel is read-only too).

## Testing and validation

Unit tests (vitest, extending the M2 files and stub patterns):

- `mcp-config`: valid stdio/sse/http entries parse; reserved name,
  malformed JSON, wrong shapes produce errors without throwing; unknown
  fields dropped.
- protocol: round-trips for the three new events and the changed events
  with/without the new optional fields.
- broker/gating (through the session manager's seams where they exist,
  else broker-level): external `mcp__` tools prompt in `approve`, are
  denied in `readOnly`, auto-approved in `acceptAll`; unknown non-MCP
  names still denied; `kube_kubectl` uses the RUN KUBECTL descriptor.
- kubectl tool: argv validation (empty, newline injection), shell quoting
  of the display command, timeout and no-subscriber fast-fail, result
  truncation; pending runs rejected on dispose.
- bridge: `GET .../tools` shape with and without a live query;
  `POST .../terminal-result` unknown-runId 404.
- terminal runner: sentinel extraction from a synthetic PTY stream
  (ANSI-colored, echoed command line included), exit-code parsing,
  capture:false path, single-post guarantee.
- preferences: new defaults and `PreferencesState` view; shortcut JSON
  parsing (valid, invalid, partial entries).
- subagents: definition wiring on/off with the preference; `Task`
  disallowed when off; parented events carry `parentCallId`.

Validation gates unchanged (AGENTS.md): `pnpm lint:fix && pnpm type:check
&& pnpm test:unit` and `pnpm build`; everything must stay green with
Claude Code absent.
