# M1 — Tool and Safety Parity: Implementation Specification

Status: **delivered** — implemented, merged, and confirmed working by the
maintainer (approved plan: [PLAN.md](./PLAN.md), issue
[#7](https://github.com/freelensapp/freelens-for-claude-extension/issues/7)).
The next milestone is specified in [M2.md](./M2.md).

Success criterion: on a subscription-authenticated Claude Code, ask
*"scale the nginx deployment to 3 replicas and restart it"* — the model
calls the mutating tools, each call pauses on an approval dialog showing
the proposed change (with a YAML diff for updates), the user approves,
the cluster changes, and after restarting Freelens the conversation
resumes where it left off.

## Scope

In scope (the M1 milestone from PLAN.md):

1. Full Kubernetes tool parity with freelens-ai-extension: the M0
   read-only trio upgraded, one new read-only tool, and six mutating
   tools (create, update, patch incl. scale, delete, pod delete/evict,
   rollout restart).
2. Human-in-the-loop approvals: `canUseTool` bridged over the HTTP/SSE
   bridge to an approval dialog with YAML preview, current-resource
   backup, and a unified diff for full updates.
3. Per-cluster permission modes: `readOnly`, `approve` (default),
   `acceptAll`.
4. Session persistence: transcript restore and Claude Code `resume`
   across app restarts; a persisted per-cluster session store.

Out of scope (deferred per PLAN.md): preferences page and the
`podLogsRequireApproval` preference (M2 — pod logs stay auto-allowed in
M1), per-cluster "always allow" lists (M2), token usage display, model
picker, retry button, tool-call cards, "Ask Claude" context menu (M2),
user MCP passthrough, subagents, Kubectl-via-Freelens-terminal (M3).
Editing tool input inside the approval dialog is also deferred: the
original extension only offers approve/deny, and M1 matches that
(`updatedInput` echoes the original input unchanged).

## Carry-over fix from M0

`session-manager.ts` returns `{ behavior: "allow", updatedInput: {} }`
from `canUseTool`. This is latent in M0 (the three read-only tools are
in `allowedTools`, so the callback never fires for them) but becomes a
real bug the moment a tool relies on the callback: `updatedInput: {}`
would erase the tool's arguments. M1 replaces the callback wholesale
(see Approval flow); the new implementation must always pass the
received `input` back as `updatedInput` when allowing.

## Protocol additions (`src/common/protocol.ts`)

New SSE event types in `SessionEventMap`:

| Event | Payload |
| --- | --- |
| `permission_request` | `{ requestId, toolName, actionTitle, input, proposedYaml, currentYaml? , diff? }` |
| `permission_resolved` | `{ requestId, behavior: "allow" \| "deny", reason?: string }` |
| `session_meta` | `{ permissionMode, resumed: boolean }` |

- `actionTitle` is a short human header in the original extension's
  style, e.g. `CREATE DEPLOYMENT`, `UPDATE SERVICE`, `DELETE POD (evict)`,
  `RESTART STATEFULSET`.
- `proposedYaml` is the tool input rendered as YAML.
- `currentYaml` is the pre-change YAML of the target resource
  (managedFields stripped), captured best-effort — a failed capture must
  never block the prompt (send `undefined`).
- `diff` is a unified diff (`diff.createTwoFilesPatch`) from
  `currentYaml` to the proposed full manifest — only for
  `kube_update_resource`, where the proposed result is known exactly.
  Patch operations show `proposedYaml` (the patch document) next to
  `currentYaml` instead; do not try to compute the effect of a merge or
  strategic-merge patch client-side.

New request bodies:

| Route | Body |
| --- | --- |
| `POST /permissions/:requestId` | `{ behavior: "allow" \| "deny" }` |
| `POST /clusters/:id/permission-mode` | `{ mode: PermissionMode }` |

`PermissionMode = "readOnly" | "approve" | "acceptAll"`.

Transcript replay: `permission_request` and `permission_resolved` are
both added to `PERSISTED_EVENTS`, so a remounted or restarted page can
render historical approvals collapsed (parity with the original's
resolved-interrupt display) and re-show a still-pending dialog.

## Approval flow (main process)

`session-manager.ts` gains a per-session permission broker:

- Read-only tool names stay in `allowedTools` (auto-allowed, no
  callback). Mutating tool names are **not** listed in `allowedTools`
  and not in `disallowedTools`, so the SDK consults `canUseTool` for
  exactly those.
- `canUseTool(toolName, input, { signal })`:
  1. Unknown tools (not ours): deny, same message as M0.
  2. Mode `readOnly`: deny mutating tools with a message telling the
     model the chat is in read-only mode.
  3. Mode `acceptAll`: allow immediately with `updatedInput: input`,
     and emit `permission_request` immediately followed by
     `permission_resolved` (`behavior: "allow"`), so the transcript
     records what auto-approval did (parity with the original's
     `bypassApprovals` rendering).
  4. Mode `approve`: capture the backup YAML (best-effort read of the
     target resource), build the diff for updates, emit
     `permission_request` with a fresh `requestId`, and return a
     promise resolved by `POST /permissions/:requestId`.
- Resolution: allow resolves `{ behavior: "allow", updatedInput: input }`;
  deny resolves `{ behavior: "deny", message: "The user denied the
  action." }` (the exact denial string the original returns to the
  model). Either way emit `permission_resolved`.
- Lifecycle: a pending request is denied (reason in
  `permission_resolved`) when the turn is interrupted, the session is
  disposed (new chat), the SDK aborts the callback via `signal`, or the
  extension deactivates. There is deliberately **no timeout** — the
  original waits indefinitely for the user, and so do we; Stop is the
  escape hatch.
- Duplicate or unknown `requestId` resolutions answer `404`; resolving
  an already-resolved request answers `409`.

Permission modes:

- Stored per cluster. `readOnly` and `approve` persist in the
  `ChatSessionStore`; `acceptAll` is **never persisted** (parity with
  the original's non-persistent `bypassApprovals`) and falls back to
  `approve` on restart.
- `POST /clusters/:id/permission-mode` switches the mode at any time;
  the new mode is emitted as `session_meta` and applies to the next
  `canUseTool` call (a pending dialog is not retroactively resolved).
- Mode changes must not require restarting the SDK query: the mode is
  read from session state inside the callback, not baked into options.

## Session persistence and resume

New `src/common/session-store.ts` — `ChatSessionStore`
(`Common.Store.ExtensionStore`, written by main, synced like the
`BridgeStore`):

```text
sessions: Record<clusterId, {
  sessionId?: string;        # Claude Code session id for resume
  permissionMode: "readOnly" | "approve";   # acceptAll not persisted
  updatedAt: string;         # ISO timestamp
}>
```

- The SDK reports the session id in the `system`/`init` message; the
  session manager captures it and writes it to the store.
- On the first `sendMessage` for a cluster after a restart, pass
  `resume: <sessionId>` in the query options. If resuming fails (the
  SDK errors because the session is gone), clear the stored id, start a
  fresh session transparently, and emit a non-fatal `error` event only
  if the fresh start also fails.
- `session_meta` is emitted to every new SSE subscriber (before the
  transcript replay) with `resumed: true` when the session was resumed.

Transcript persistence — the rendered transcript must survive restarts
without bloating the synced store:

- The in-memory `transcript` buffer (`PERSISTED_EVENTS`) is additionally
  written as JSON to `transcript.json` inside the per-cluster scratch
  directory (the existing `cwd`), debounced (e.g. 500 ms after the last
  event) and flushed on dispose/deactivate.
- On session construction, load `transcript.json` if present so SSE
  replay restores the conversation after an app restart — the M0 replay
  mechanism then works unchanged.
- `DELETE /clusters/:id/session` (new chat) deletes `transcript.json`,
  clears the stored `sessionId`, and keeps the persisted permission
  mode.

System prompt update (`session-manager.ts`): the M0 "read-only, cannot
modify" wording is replaced by mode-aware text — the model operates on
cluster `<name>` through `kube_` tools; mutating tools exist but every
mutation requires explicit user approval (or is denied in read-only
mode); it should prefer reads to discover state before proposing
mutations, and must never retry a denied action without being asked.

## Tools

Parity mapping against the original extension's 12 tools (research on
`freelens-ai-extension@main`, `src/renderer/business/agent/tools/`):

| Original tool | This extension |
| --- | --- |
| `getNamespaces` | covered by `kube_resources` (kind `Namespace`) |
| `getClusterVersion` | `kube_cluster_version` (new) |
| `getEventsForNamespace` | `kube_warning_events` (M0) |
| `listKubernetesResources` | `kube_resources` (M0, upgraded) |
| `getKubernetesResource` | `kube_resources` (M0, upgraded) |
| `getPodLogs` | `kube_pod_logs` (M0, upgraded) |
| `createKubernetesResource` | `kube_create_resource` (new) |
| `updateKubernetesResource` | `kube_update_resource` (new) |
| `patchKubernetesResource` (incl. scale/resize subresource) | `kube_patch_resource` (new) |
| `deleteKubernetesResource` (delete / force / finalize) | `kube_delete_resource` (new) |
| `deletePod` (evict / force / finalizers) | `kube_delete_pod` (new) |
| `restartKubernetesResource` | `kube_rollout_restart` (new) |

Read-only set (in `allowedTools`, auto-allowed): `kube_resources`,
`kube_pod_logs`, `kube_warning_events`, `kube_cluster_version`.
Mutating set (via `canUseTool`): the other six. `mcp-server.ts` exports
both name lists so the session manager and tests share one source of
truth.

All tools keep the M0 conventions: zod schemas with described fields, a
narrow client interface per tool for testability, errors returned as
explanatory text results (never thrown), output as
`content: [{ type: "text", text }]`.

### Upgrades to existing tools

`kube_resources` — two new optional inputs (parity with the original's
list/get):

- `fields?: string[]` — JSONPath-subset selectors applied to each
  returned object, e.g. `metadata.name`, `spec.containers[*].image`,
  `status.conditions[0].type`, `metadata.labels['app.kubernetes.io/name']`.
  Implement `selectFields(object, selectors)` in `kube-format.ts`
  supporting: dot-separated keys, `[*]` wildcard over arrays, numeric
  index (including negative), and bracketed quoted keys for dotted
  label keys; matches are copied into a fresh object preserving nested
  structure. A malformed selector returns a readable error to the
  model. Tolerate a leading `$.` and a `{...}` wrapper.
- `includeManagedFields?: boolean` (default `false`) — opt back in to
  `metadata.managedFields`.

`kube_pod_logs` — two new optional inputs:

- `previous?: boolean` — logs of the last terminated container instance
  (the CrashLoopBackOff case); a "previous terminated container not
  found" API error becomes a friendly message.
- `timestamps?: boolean` — RFC3339 timestamp prefix per line.

`kube_warning_events` stays as in M0 (already matches the original).

### New read-only tool

`kube_cluster_version` — no inputs. GET the API server `/version`
endpoint (`@kubernetes/client-node` `VersionApi.getCode()`); return a
short text summary: gitVersion, major/minor, platform, buildDate.

### New mutating tools

1. `kube_create_resource` — input `{ manifest: object }` (the full
   resource as a JSON object; `apiVersion`, `kind`, `metadata.name`
   required, validated up front with a clear error). Strips any
   client-supplied `metadata.managedFields`, then
   `KubernetesObjectApi.create`. Approval title `CREATE <KIND>`;
   `proposedYaml` is the manifest.
2. `kube_update_resource` — input `{ manifest: object }` (full
   replacement manifest). Reads the current object first (this is also
   the approval backup), carries over `metadata.resourceVersion` when
   the manifest omits it, then `KubernetesObjectApi.replace`. Approval
   title `UPDATE <KIND>` with `currentYaml` and the computed `diff`.
3. `kube_patch_resource` — input `{ apiVersion, kind, namespace?, name,
   patch: object, subresource? }`. Without `subresource`:
   `KubernetesObjectApi.patch` with JSON merge patch
   (`application/merge-patch+json`) — the original's default. With
   `subresource` (normalized lowercase, e.g. `scale`, `resize`,
   `status`): PATCH `<resource-url>/<subresource>` with strategic merge
   patch (`application/strategic-merge-patch+json`) so named
   container-array entries merge — this is how **scale** works (parity:
   the original has no dedicated scale tool; the tool description must
   tell the model to patch subresource `scale` with
   `{ spec: { replicas: n } }` to scale). Implementation hint: the
   generic subresource PATCH is a raw `fetch` built from
   `KubeConfig.applyToFetchOptions` against the resource URI resolved
   via the discovery client, since `KubernetesObjectApi.patch` does not
   take a subresource. Approval title `PATCH <KIND>` (or
   `PATCH <KIND> (<subresource>)`), `currentYaml` backup, `proposedYaml`
   is the patch document.
4. `kube_delete_resource` — input `{ apiVersion, kind, namespace?,
   name, mode? }`, `mode` in `delete` (default) | `force_delete` |
   `force_finalize` (parity with the original's modes):
   - `delete`: `KubernetesObjectApi.delete` with defaults;
   - `force_delete`: delete with `gracePeriodSeconds: 0`,
     `propagationPolicy: Background`;
   - `force_finalize`: merge-patch `{ metadata: { finalizers: [] } }` —
     documented in the schema description as a last resort for
     resources stuck Terminating.
   Approval title `DELETE <KIND> (<mode>)`, `currentYaml` backup.
5. `kube_delete_pod` — input `{ namespace, name, mode }`, `mode`
   **required**, in `evict` | `force_delete` | `delete_with_finalizers`:
   - `evict`: create a `policy/v1` `Eviction` for the pod
     (`CoreV1Api` eviction call) — honors PodDisruptionBudgets, the
     preferred mode for voluntary disruption;
   - `force_delete`: delete with grace period 0 — for pods on
     unreachable or NotReady nodes;
   - `delete_with_finalizers`: clear the pod's finalizers then delete —
     last resort for pods stuck Terminating.
   Approval title `DELETE POD (<mode>)`, `currentYaml` backup.
6. `kube_rollout_restart` — input `{ kind, namespace, name }`, `kind`
   restricted to `Deployment` | `DaemonSet` | `StatefulSet` (reject
   others up front, parity with the original's RESTARTABLE_KINDS).
   Strategic-merge patch setting
   `spec.template.metadata.annotations["kubectl.kubernetes.io/restartedAt"]`
   to the current ISO timestamp — exactly what `kubectl rollout
   restart` does. Approval title `RESTART <KIND>`, `currentYaml`
   backup.

### Kube client additions (`kube-client.ts`)

Extend `KubeClient` with the API surface the new tools need:
`VersionApi`, and whatever the eviction and subresource-PATCH paths
require (`CoreV1Api` already present; add helpers rather than leaking
raw config where a typed client exists). Keep the per-cluster cache and
the throw-descriptive-errors contract unchanged.

## Renderer

`permission-dialog.tsx` — rendered inline in the transcript at the
position of the `permission_request` event (parity with the original's
interrupt component, not a modal):

- warning-styled header with `actionTitle`;
- "Action details" block: `diff` rendered as a diff-highlighted code
  block when present, otherwise `proposedYaml`;
- "Current resource (backup)" block, collapsed by default, showing
  `currentYaml` when present — presented as the restorable backup;
- Approve / Deny buttons while pending; after `permission_resolved`
  the card collapses to a one-line resolved state (approved/denied,
  click to expand details). Buttons POST to
  `/permissions/:requestId` via a new `resolvePermission` method on
  `BridgeClient`.

`chat-view.tsx`:

- reducer handles the three new event types; `permission_request`
  appends a chat item, `permission_resolved` marks it resolved,
  `session_meta` sets the mode (and can show a subtle "conversation
  resumed" notice when `resumed` is true);
- the input stays enabled while a permission request is pending (the
  user must be able to hit Stop / New chat), but sending is still
  blocked while `working`;
- status strip gains a permission-mode selector (`Read-only` /
  `Approve` / `Accept all` labels) wired to
  `POST /clusters/:id/permission-mode`; `acceptAll` gets a visually
  distinct warning state.

`bridge-client.ts` gains `resolvePermission(requestId, behavior)` and
`setPermissionMode(clusterId, mode)`.

## Testing and validation

Unit tests (vitest, extending the M0 test files and stub pattern):

- permission broker: approve resolves with the original input as
  `updatedInput`; deny returns the exact denial message; `readOnly`
  denies without emitting a request; `acceptAll` emits
  request+resolved pair; interrupt/dispose denies pending requests;
  double-resolution is rejected.
- bridge: `POST /permissions/:id` happy path, 404 unknown, 409 already
  resolved; `POST /permission-mode` validation.
- protocol: encode/decode round-trip for the new event types.
- `selectFields`: dot paths, `[*]`, numeric and negative index, quoted
  bracket keys, malformed-selector error text.
- each new tool: schema validation, mode/kind restrictions
  (`kube_rollout_restart` kind rejection, `kube_delete_pod` required
  mode), managedFields stripping on create, resourceVersion carry-over
  on update, patch content types by subresource (mock clients).
- session store: round-trip, `acceptAll` never persisted; resume option
  passed on first message after restore (mock SDK factory seam if
  needed).

Validation gates unchanged (AGENTS.md): `pnpm lint:fix &&
pnpm type:check && pnpm test:unit` and `pnpm build`; the integration
test must stay green with Claude Code absent.
